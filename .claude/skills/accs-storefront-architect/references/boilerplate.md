# AEM Boilerplate Commerce — Reference

Adobe's officially supported starter codebase for Edge Delivery Services (EDS) storefronts wired to Adobe Commerce (PaaS) or Adobe Commerce as a Cloud Service (ACCS / Optimizer).
Lives at `github.com/hlxsites/aem-boilerplate-commerce`; bundles Commerce drop-in NPM packages, EDS scripts/blocks, and configuration glue so storefronts ship without hand-wiring every Commerce integration.

## What is the commerce boilerplate

The Commerce boilerplate is a superset of the generic EDS boilerplate (`aem-boilerplate`) with Commerce-specific additions layered on top of standard EDS plumbing (`scripts/aem.js`, `blocks/`, `styles/`, document authoring via DA.live or Google Docs/SharePoint, preview at `*.aem.page`, production at `*.aem.live`).

What it adds on top of generic EDS:

- **Commerce drop-in NPM packages** (`@dropins/storefront-*`) — pre-built, versioned UI components for cart, checkout, PDP, product discovery, auth, account, recommendations, wishlist, order, returns, requisition lists, quote management. These ship as framework-free Web Components powered by a shared event bus and GraphQL client.
- **`scripts/commerce.js`** — Commerce-specific orchestration (GraphQL fetchers, config wiring, drop-in bootstrapping). Sits beside `scripts/aem.js`.
- **`scripts/initializers/`** — One init file per drop-in domain. Each initializer calls `setEndpoint`, fetches placeholders, mounts the drop-in.
- **Tool packages** — `@dropins/tools` (event bus, initializer, GraphQL client), `@adobe/adobe-client-data-layer`, `@adobe/magento-storefront-event-collector`, `@adobe/magento-storefront-events-sdk` (events feed Live Search, Recommendations, Sensei).
- **Commerce blocks** — ~30 blocks under `blocks/` that host drop-ins (see Blocks Reference below).
- **Universal Editor instrumentation** — Co-located `blocks/{name}/_{name}.json` files composed by a pre-commit hook into root `component-*.json` manifests.
- **`config.json` schema** — Endpoints, headers, store codes, analytics IDs, plugin config (see Configuration model).

Forks like Citisignal are demo/showcase storefronts and may lag the canonical repo — do not use them as project starting points.

## Getting started

Prereqs:

- Node.js + npm (Node 22+ recommended for the AI agent skills; the boilerplate itself works on supported LTS).
- A GitHub repo created from the boilerplate template and connected to AEM Code Sync (per Adobe's "Create a storefront" guide).
- Document Authoring (DA.live) workspace initialized OR Google Drive/SharePoint mountpoint configured.
- An Adobe Commerce backend — either **Adobe Commerce (PaaS / on-prem / Cloud)** with the Catalog Service / Live Search / Payment Services / Storefront Services SaaS add-ons enabled, OR **Adobe Commerce as a Cloud Service (ACCS) / Commerce Optimizer** instance.
- `x-api-key`, `Magento-Environment-Id`, store/website/view codes from the backend.
- (Optional) AEM CLI globally: `npm install -g @adobe/aem-cli`.
- (Optional, for AI agent skills) `@adobe/aio-cli` + `aio-cli-plugin-commerce`.

Quick start:

```bash
git clone <your-repo-from-template>
cd <repo>
npm install         # also runs install:dropins via lifecycle hook
npm start           # serves at http://localhost:3000 via aem-cli
```

`npm install` triggers `install:dropins` (postinstall hook) which fetches and links the `@dropins/storefront-*` packages. Do not delete this hook.

Deployment is implicit — pushing to GitHub publishes via Code Sync:

- Preview: `https://<branch>--<repo>--<owner>.aem.page`
- Production: `https://<branch>--<repo>--<owner>.aem.live`

No separate build/deploy step.

Backend wiring checklist (in `config.json` or via Config Service):

- Set `commerce-endpoint` (always — points to catalog/optimizer GraphQL).
- Set `commerce-core-endpoint` only for ACCS/Optimizer split-backend setups (core/checkout GraphQL). For pure Adobe Commerce Cloud single-backend, omit it.
- Set headers: `Store`, `Magento-Store-Code`, `Magento-Store-View-Code`, `Magento-Website-Code`, `x-api-key`, `Magento-Environment-Id`.
- Populate analytics keys (AEP IMS Org, Datastream, environment-id, store IDs/names) so the event collector fires correctly.

## Configuration model

Three sources, in priority order:

1. **`config.json` at repo root** — Local development override. Served at `/config.json` by `aem-cli` and overrides anything from the Config Service if committed. **Do NOT commit to `main` if using the Configuration Service** — repo files win over the service.
2. **Configuration Service** — `https://admin.hlx.page/config/{ORG}/sites/{SITE}/public.json` (POST with auth to update). Used for preview/production. Holds `default-site.json`, `default-query.yaml`, `default-sitemap.yaml`.
3. **Runtime / drop-in initializers** — `scripts/initializers/*.js` can override endpoints, models, language definitions per drop-in at boot time.

```json
    "default": {

      "commerce-assets-enabled": true,
      "headers": {
        "all": {
          "Store": "default"
        },
        "cs": {
          "Magento-Store-Code":      "main_website_store",
          "Magento-Store-View-Code": "default",
          "Magento-Website-Code":    "base",
          "x-api-key":               "<key>",
          "Magento-Environment-Id":  "<env-uuid>"
        }
      },
      "analytics": {
        "aep-ims-org-id":     "<org>@AdobeOrg",
        "aep-datastream-id":  "<uuid>",
        "base-currency-code": "USD",
        "environment":        "Production",
        "environment-id":     "<uuid>",
        "store-code":         "main_website_store",
        "store-id":           "1",
        "store-name":         "Main Store",
        "store-url":          "https://www.example.com",
        "store-view-code":    "default",
        "store-view-id":      "1",

        "website-code":       "base",

      },
      "plugins": {
        "picker": {
          "rootCategory": "<category-uid>"
        }
      }

```

### Complete config keys (verbatim)

Endpoints:

- `commerce-core-endpoint` — GraphQL endpoint for Commerce core / PaaS (checkout, customer, order). Required only for ACCS / split backends.
- `commerce-endpoint` — GraphQL endpoint for catalog / Optimizer (PDP, PLP, search).
- `commerce-assets-enabled` — Boolean; toggles AEM Assets integration for product imagery.

Headers (`headers.all` — applied to every request):

- `Store`

Headers (`headers.cs` — Commerce Services / catalog requests):

- `Magento-Store-Code`
- `Magento-Store-View-Code`
- `Magento-Website-Code`
- `x-api-key`
- `Magento-Environment-Id`

Analytics: `aep-ims-org-id`, `aep-datastream-id`, `base-currency-code`, `environment`, `environment-id`, `store-code`, `store-id`, `store-name`, `store-url`, `store-view-code`, `store-view-id`, `store-view-name`, `website-code`, `website-id`, `website-name`.

Plugins: `plugins.picker.rootCategory` — root category for the in-editor product/category picker.

### Service-side configs

`default-site.json` keys: `version`, `code` (owner/repo/source), `content.source.url`, `content.source.type`, `folders`, `cdn.live.host`, `cdn.preview.host`, `headers`, `public.default` (mirrors `config.json` schema), `robots.txt`, `access.admin.roles`, `access.requireAuth`.

`default-query.yaml` — defines two indices: `sitemap` → `/sitemap.json` (excludes drafts and enrichment paths); `enrichment` → `/enrichment/enrichment.json` (includes enrichment paths).

`default-sitemap.yaml`:
```yaml
sitemaps:
  default:
    source: /sitemap.json
    destination: /sitemap-content.xml
```

### Secret handling

`x-api-key` is sent client-side to Commerce SaaS endpoints — it is a scoped catalog read key, not a server secret. Treat it as low-sensitivity but rotate when leaked. For higher-trust calls, use the Configuration Service so the key lives in the published `public.json` rather than the public repo. Authoring credentials, Adobe I/O service-account keys, and ACCS admin tokens must never appear in `config.json` or any committed file.

### Drop-in initializer pattern

Every drop-in is mounted via `scripts/initializers/{domain}.js`:

```javascript
import { initializers } from '@dropins/tools/initializer.js';
import { initialize, setEndpoint } from '@dropins/storefront-{feature}/api.js';
import { fetchPlaceholders } from '../aem.js';
import { CORE_FETCH_GRAPHQL } from '../commerce.js';

await initializeDropin(async () => {
  setEndpoint(CORE_FETCH_GRAPHQL);
  const labels = await fetchPlaceholders('placeholders/{feature}.json');
  return initializers.mountImmediately(initialize, { langDefinitions: labels });
})();
```

`fetchPlaceholders` reads label spreadsheets (per-feature `placeholders/{feature}.json`) so copy is authorable without code changes.

## Blocks reference

Every Commerce block is a thin EDS block (`decorate(block)` default export) that hosts one or more drop-ins. Document-authoring contract is a single block table with the block name as the table title; rows become `readBlockConfig()` key/value pairs (kebab-case keys, string values). Blocks without table content render with defaults.

### Discovery & PDP

**Product List Page** (`blocks/product-list-page`) — Searchable, filterable, sortable PLP. Hosts `storefront-product-discovery`, `storefront-wishlist`, `storefront-requisition-list`, `storefront-cart`. Authoring: empty block; placement on a category-template page. Features: search, filters, sort, pagination, wishlist toggles.

**Product Details** (`blocks/product-details`) — PDP showcase. Hosts `storefront-pdp`, `storefront-wishlist`, `storefront-requisition-list`. Authoring: empty block on a product-template page. Modern architecture uses independent containers (`ProductHeader`, `ProductPrice`, `ProductGallery`, etc.) rather than a single monolithic container with slots — arrange in any order. Features: configurable options, pricing, add-to-cart, wishlist toggle.

**Product Recommendations** (`blocks/product-recommendations`) — AI-powered recommendation rail. Hosts `storefront-cart`, `storefront-recommendations`, `storefront-wishlist`. Authoring: block table with placement / unit-id config. Multiple page-type contexts (PDP, cart, PLP, home).

### Cart

**Commerce Cart** (`blocks/commerce-cart`) — Full cart page. Hosts `storefront-cart`, `storefront-wishlist`, `storefront-quote-management`.
Notable `readBlockConfig` options (all string-typed in document tables): `hide-heading` (default `false`); `max-items`; `hide-attributes` (comma-separated list of attribute codes to suppress); `enable-item-quantity-update` (default `false`); `enable-item-remove` (default `true`); `enable-estimate-shipping` (default `false`). Features: item management, coupons, gift options slot, move-to-wishlist.

**Commerce Mini Cart** (`blocks/commerce-mini-cart`) — Header dropdown summary. Hosts `storefront-cart`. Authoring: usually auto-blocked or placed in nav.

### Checkout

**Commerce Checkout** (`blocks/commerce-checkout`) — Full checkout flow. Hosts `storefront-order`, `storefront-checkout`. Customization is event-driven (no significant document config options) — listen on the event bus for step transitions.

### Auth

**Commerce Login** / **Commerce Create Account** / **Commerce Confirm Account** / **Commerce Forgot Password** / **Commerce Create Password** — Each hosts `storefront-auth`. Thin wrappers — no document config.

### Account

**Commerce Account Header** — Customer name display + logout. Tools-only.
**Commerce Account Sidebar** — Account nav with active-state. Hosts `storefront-account`.
**Commerce Addresses** — Address-book CRUD. Hosts `storefront-account`.
**Commerce Customer Information** — Profile editing. Hosts `storefront-account`.
**Commerce Customer Details** — Customer info display rendered inside an order page. Hosts `storefront-order`.

### Orders

**Commerce Orders List** — Order history. Hosts `storefront-account`.
**Commerce Search Order** — Guest order lookup. Hosts `storefront-auth`, `storefront-order`.
**Commerce Order Header** — Order ID / date / status badge. Tools-only.
**Commerce Order Status** — Fulfillment tracking detail. Hosts `storefront-order`.
**Commerce Order Product List** — Line items with reorder. Hosts `storefront-order`, `storefront-cart`.
**Commerce Order Cost Summary** — Subtotal/tax/shipping/discount/total. Hosts `storefront-order`.
**Commerce Shipping Status** — Shipment tracking. Hosts `storefront-order`.

### Returns

**Commerce Returns List** — Return-request history. Hosts `storefront-order`.
**Commerce Create Return** — Return-request form. Hosts `storefront-order`.
**Commerce Order Returns** — Return detail in the context of a specific order. Hosts `storefront-order`.
**Commerce Return Header** — Return ID, date, status. Tools-only.

### Other

**Commerce Gift Options** — Gift message + wrapping + receipt selectors, typically slotted into cart/checkout. Hosts `storefront-cart`.
**Commerce Wishlist** — Saved-items page with move-to-cart. Hosts `storefront-cart`, `storefront-pdp`, `storefront-wishlist`, `storefront-auth`.

### Integration architecture

Lifecycle per block: **server-side rendering → client-side decoration → drop-in initialization → rendering → event handling**.

Block decoration patterns:
- **Single drop-in wrapper** (Login, Create Account, etc.): `decorate(block)` calls `render()` from the drop-in module on the block element.
- **Multi-drop-in coordinator** (Cart, Wishlist, PDP): block reads config, builds DOM scaffolding with `createContextualFragment`, mounts each drop-in into its own container, wires shared state via `events.on(...)`.

Shared event bus topics include `cart/data`, `cart/updated`, `auth/signedin`, `pdp/data`, etc. — every drop-in publishes/subscribes through `@dropins/tools/event-bus.js`.

## Customizing blocks

Three layers — apply in order, only escalate as needed.

### 1. CSS

Edit `blocks/{name}/{name}.css`. Use the design-token CSS variables — do not inline colors/fonts: `var(--spacing-small)` … `var(--spacing-large)`; `var(--color-brand-500)`, `var(--color-neutral-100)`, etc.; `var(--type-body-1-default-font)`.

### 2. JavaScript behavior

Edit `blocks/{name}/{name}.js`. Standard skeleton:

```javascript
import { readBlockConfig } from '../../scripts/aem.js';

export default async function decorate(block) {
  const {
    'custom-option': customOption = 'default',
    'max-items': maxItems = '10',
    'enable-feature': enableFeature = 'false',
  } = readBlockConfig(block);
  // build DOM, mount drop-ins, attach event listeners
}
```

### 3. Drop-in customization via slots and events

**Slots** intercept rendering at named extension points. Pass a function per slot in `render()` config; it receives a `ctx` containing the target element to mutate.

```javascript
import { CartSummaryList } from '@dropins/storefront-cart/containers/CartSummaryList.js';
import { render as provider } from '@dropins/storefront-cart/render.js';

await provider.render(CartSummaryList, {
  slots: {
    Footer: (ctx) => {
      const element = document.createElement('div');
      element.classList.add('custom-footer');
      ctx.appendChild(element);
    },
  },
})($list);
```

**Events** — listen for state via the shared bus instead of subclassing:

```javascript
import { events } from '@dropins/tools/event-bus.js';

events.on('cart/data', (cartData) => {
  // react to cart updates
});
```

### DOM creation rules

Multiple elements at once — use `createContextualFragment`:

```javascript
const fragment = document.createRange().createContextualFragment(`
  <div class="cart__wrapper">
    <div class="cart__list"></div>
    <div class="cart__summary"></div>
  </div>
`);
const $list = fragment.querySelector('.cart__list');
block.appendChild(fragment);
```

Single element — use `document.createElement`.

### Override vs extend decision

- **Extend** (preferred) — add slots, attach event listeners, layer CSS. Survives drop-in upgrades.
- **Override** — fork the drop-in container or rewrite the block to call lower-level APIs. Used when slots cannot reach the needed extension point. Requires re-validation on every drop-in upgrade.

Simple auth/account blocks are "thin wrappers around drop-ins with no document-based configuration options" — extend via slots/events; reach into the drop-in directly only when slots don't expose the needed hook.

### Authoring control via config

Expose merchant-controllable knobs through document tables rather than code constants — read them with `readBlockConfig(block)` and surface them in the block's `_{name}.json` so they're editable in DA.live / UE.

### Empty-state customization

Each drop-in container exposes empty-state slots (e.g., cart's `EmptyCart`); customize copy and CTAs there rather than gating on data presence in the block.

## Universal Editor integration

Distributed instrumentation: every block carries its own UE definition next to its source.

```
aem-boilerplate-commerce/
├── component-definition.json    # GENERATED
├── component-filters.json       # GENERATED
├── component-models.json        # GENERATED
├── models/
│   └── _section.json            # registry of available blocks per section
└── blocks/
    ├── accordion/_accordion.json
    ├── hero/_hero.json
    ├── commerce-cart/_commerce-cart.json
    ├── product-details/_product-details.json
    ├── product-recommendations/_product-recommendations.json
    └── ...
```

Authoring flow:
1. Author each block's UE definition in `blocks/{name}/_{name}.json` — declares field models (label, type, value formats), filters (which sections allow this block), and the block's component definition.
2. A **pre-commit hook composes** these into root-level `component-definition.json`, `component-filters.json`, `component-models.json` — UE consumes those.
3. Add the block's component ID to `models/_section.json` so it appears in the section picker.
4. Blocks lacking a `_{name}.json` show as **"(no definition)"** in the UE content tree — instant visual signal that instrumentation is missing.

What's UE-editable: block-level `readBlockConfig` keys (exposed as form fields), slot content (rich text, image, link fields), section-level container settings, page-level metadata (when modeled). Commerce drop-ins themselves are not UE-instrumented — only the surrounding host block's config and slot content are authorable.

## Updates & versioning

Two independent update paths.

### 1. NPM drop-in updates (primary, recommended)

Drop-ins follow **semver** — minor/patch are non-breaking by contract; major versions may break.

```bash
npm list @dropins/storefront-*
npm update @dropins/storefront-*
```

Key packages: `@dropins/storefront-cart`, `@dropins/storefront-checkout`, `@dropins/storefront-pdp`, `@dropins/storefront-product-discovery`, `@dropins/storefront-account`, `@dropins/storefront-auth`, `@dropins/storefront-order`, `@dropins/storefront-recommendations`, `@dropins/storefront-wishlist`, `@dropins/storefront-requisition-list`, `@dropins/storefront-quote-management`, `@dropins/tools` (shared event bus / GraphQL client / initializer), B2B-flavored variants (requisition lists, quote management, etc.).

Automation: the included `.github/workflows/update-dropins.yml` GitHub Action opens a weekly PR bumping all drop-in packages.

### 2. Selective boilerplate merges (advanced, optional)

> "You are under no obligation to pull upstream boilerplate changes into your project."

Most teams generate from the template and never sync upstream. When you do sync — to pick up improved initializer patterns, new boilerplate blocks, or refactors — cherry-pick or selectively merge from `hlxsites/aem-boilerplate-commerce`. Focus areas: `scripts/commerce.js`, `scripts/initializers/`, `blocks/` (only the blocks you actually use). Skip changes to files you've customized; reapply manually if needed.

### Suite releases

Adobe periodically publishes "Suite" snapshots that pin specific drop-in versions to specific boilerplate code. Use them as **validated starting points** for new projects, not as upgrade targets for existing projects (which should track drop-in NPM versions independently).

### Mid-2025 architecture refactor (heads-up for early adopters)

- **Before:** all logic lived in `scripts/scripts.js`.
- **After:** Commerce logic was extracted into `scripts/commerce.js` and the `@dropins/*` packages.

This split reduced merge conflicts and isolated business logic in versioned, upgradable packages. Projects forked before this refactor should plan a one-time migration to align with the new layout.

### PDP container migration

- **Deprecated:** monolithic `ProductDetails` container with slots.
- **Current:** independent containers (`ProductHeader`, `ProductPrice`, `ProductGallery`, `ProductOptions`, `ProductActions`, `ProductDescription`, etc.) mounted side-by-side.

Benefits: layout freedom (arrange containers in any order), direct event-bus data access, removal of complex CSS slot workarounds. Migrate by replacing the single `ProductDetails` render call with per-container renders inside the block's DOM scaffolding.

### Cadence & testing

- Review monthly or quarterly; deploy during low-traffic windows.
- Maintain e2e tests (Cypress or similar) on critical journeys: browse → PDP → cart → checkout → order detail.
- After each drop-in bump, verify slot signatures haven't changed (slot `ctx` shapes are part of the public API but can shift across major versions).

### Troubleshooting

- **Post-install failures:** clear npm cache (`npm cache clean --force`), verify Node version, re-run `npm install`.
- **Merge conflicts on upstream sync:** keep your customizations; only adopt upstream changes that touch files you haven't modified.
- **Breaking-change surprises:** read the changelog (`hlxsites/aem-boilerplate-commerce` issues with `changelog` label) before bumping a major version.

## AI agent skills

Adobe ships an `aio-cli` plugin that installs coding-agent skills for the boilerplate. Six skills:

1. **Project Manager** — phased task breakdown, delivery planning.
2. **Researcher** — API lookup, documentation search.
3. **Block Developer** — EDS block creation patterns.
4. **Drop-in Developer** — container + slot customization patterns.
5. **Content Modeler** — block-table structure design (document-authoring contract).
6. **Tester** — browser verification, Core Web Vitals checks.

Prereqs:
- Node.js 22+
- A boilerplate project
- `npm install -g @adobe/aio-cli`
- `aio plugins:install https://github.com/adobe-commerce/aio-cli-plugin-commerce`

Install interactively:
```bash
aio commerce extensibility tools-setup
```

Non-interactive:
```bash
aio commerce extensibility tools-setup \
  --starter-kit aem-boilerplate-commerce \
  --agent Cursor \
  --package-manager npm
```

Skills land in the agent-specific directory: Cursor → `.cursor/skills/`; Claude Code → `.claude/skills/`; GitHub Copilot → `.github/skills/`; Windsurf → `.windsurf/skills/`; Gemini CLI → `.gemini/skills/`.

Files generated: `AGENTS.md` (session-level agent instructions); one file per skill in the chosen agent's skills directory; MCP configuration entries for live documentation access.
